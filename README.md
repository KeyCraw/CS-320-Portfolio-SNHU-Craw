## Testing Approach

This portfolio reflects a requirement-driven testing strategy. Each validation rule specified in the project requirements was translated directly into unit tests. Both positive and negative test cases were created to verify correct behavior and proper error handling.

Testing practices included:

- Validation of input constraints  
- Boundary testing for string length limits  
- Duplicate identifier detection  
- Exception handling verification  
- Defensive testing against null and invalid values  

Automated testing ensures consistent verification and strengthens software reliability.

## Software Quality and Validation

All implementations enforce strict validation rules to maintain data integrity and prevent invalid program states. Defensive programming techniques were applied to ensure that incorrect inputs are rejected and that objects cannot enter inconsistent states.

Quality principles demonstrated:

- Encapsulation  
- Immutable identifiers  
- Controlled state updates  
- Exception-based validation  
- Predictable system behavior

## Module Eight Reflection

### How can I ensure that my code, program, or software is functional and secure?

I ensure that my software is functional and secure by using structured unit testing and defensive programming techniques. By testing both valid and invalid inputs, I confirm that the system behaves correctly under normal conditions and safely handles unexpected data. Enforcing input validation and preventing invalid object states reduces potential vulnerabilities and improves reliability. Automated testing also allows repeated verification after modifications, ensuring continued stability.

### How do I interpret user needs and incorporate them into a program?

Interpreting user needs begins with analyzing the functional requirements and translating them into measurable behaviors within the program. Each requirement was converted directly into one or more unit tests. This requirement-driven approach ensures that the final implementation reflects user expectations and that each rule can be verified through automated testing.

### How do I approach designing software?

I approach software design by first identifying the core entities and services required by the system. I focus on clear class responsibilities, encapsulation, and validation to maintain consistent and reliable behavior. Designing with testing in mind encourages modular architecture and simpler, maintainable code. This disciplined approach helps prevent technical debt and supports long-term scalability.
